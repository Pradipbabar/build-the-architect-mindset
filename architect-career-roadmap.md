## Phase 1: Software Development Foundation (12-18 months)

**Goal**: Be able to build production-grade services independently in Go.

**Why this comes first**: You cannot architect systems you cannot build. This is non-negotiable.

### What to Learn

#### 1. Master Go (4-6 months)

Not "basic understanding." MASTER it.

**Core concepts** (in this order):
- Data structures & algorithms in Go
- Concurrency: goroutines, channels, sync primitives, context
- Error handling patterns (not just `if err != nil`)
- Testing: unit tests, table-driven tests, mocks, benchmarks
- Interfaces & composition
- Package design & project structure

**Resources** (free/cheap):
- [ ] Read: "The Go Programming Language" (Donovan & Kernighan) — work through EVERY example
- [ ] Complete: [Tour of Go](https://go.dev/tour/) — don't skip exercises
- [ ] Build: [Gophercises](https://gophercises.com/) — 20 exercises, do all of them
- [ ] Read: "Learning Go" by Jon Bodner — focus on chapters 5-13
- [ ] Study: Effective Go (official docs)

**Practice projects** (must build these yourself, NO AI):
1. **CLI tool**: Build a log parser that reads files, filters by pattern, outputs stats (learn file I/O, string manipulation)
2. **HTTP API**: Build a REST API for a todo list with persistence (learn HTTP, JSON, database basics)
3. **Concurrent system**: Build a web crawler that follows links with rate limiting (learn goroutines, channels, sync)
4. **Testing**: Write comprehensive tests for all above projects (aim for >80% coverage)

**Success criteria**:
- Can build any of these from scratch without AI
- Can explain WHY your code works, not just THAT it works
- Can review others' Go code and spot issues
- Comfortable with goroutines, channels, and error handling

#### 2. Database Design & SQL (2-3 months)

You can't design systems without understanding data.

**Core concepts**:
- Relational model: normalization, denormalization, relationships
- SQL: queries, joins, aggregations, subqueries, window functions
- Indexes: when/why/how to use them
- Transactions: ACID, isolation levels
- Performance: query optimization, explain plans

**Resources**:
- [ ] Course: [SQLBolt](https://sqlbolt.com/) — interactive, free
- [ ] Read: "SQL Performance Explained" by Markus Winand
- [ ] Practice: [Mode SQL Tutorial](https://mode.com/sql-tutorial/) — real datasets

**Practice projects**:
1. Design a database schema for an e-commerce system (users, products, orders, inventory)
2. Write queries for common patterns (top N, running totals, user analytics)
3. Optimize slow queries (use EXPLAIN, add indexes)
4. Add database layer to your todo API from above

**Success criteria**:
- Can design normalized schemas
- Can write complex SQL queries without googling basic syntax
- Understand when to use different index types
- Know trade-offs between normalization and performance

#### 3. APIs & Integration Patterns (2-3 months)

**Core concepts**:
- REST: design principles, HTTP methods, status codes, versioning
- gRPC: protocol buffers, streaming, error handling
- Authentication & authorization: JWT, OAuth, API keys
- API design: pagination, filtering, error responses, rate limiting

**Resources**:
- [ ] Read: "REST API Design Rulebook" by Mark Masse
- [ ] Study: [Google API Design Guide](https://cloud.google.com/apis/design)
- [ ] Study: [gRPC Go tutorial](https://grpc.io/docs/languages/go/)

**Practice projects**:
1. Build a REST API with proper error handling, validation, auth
2. Build the same API in gRPC
3. Add rate limiting, logging, metrics
4. Write API documentation (OpenAPI/Swagger)

**Success criteria**:
- Can design clean, consistent APIs
- Understand REST vs gRPC trade-offs
- Can implement auth/authz
- Know when to use different patterns

#### 4. Distributed Systems Basics (3-4 months)

**Core concepts**:
- CAP theorem & trade-offs
- Consistency models
- Message queues & async patterns
- Service-to-service communication
- Observability: logging, metrics, tracing

**Resources**:
- [ ] Read: "Designing Data-Intensive Applications" by Martin Kleppmann (chapters 1-9) — THIS IS CRITICAL
- [ ] Course: MIT 6.824 lectures on YouTube (focus on first 10 lectures)
- [ ] Read: [Google SRE Book](https://sre.google/sre-book/table-of-contents/) — chapters on monitoring, toil, SLOs

**Practice projects**:
1. Build two services that communicate via HTTP and message queue (use Redis/RabbitMQ)
2. Add retry logic, circuit breakers, timeouts
3. Add observability: structured logging, Prometheus metrics, basic tracing
4. Cause failures and see how system behaves

**Success criteria**:
- Understand CAP theorem and can explain trade-offs
- Can design for failure (retries, timeouts, circuit breakers)
- Know when to use sync vs async communication
- Can instrument code for observability

### Phase 1 Summary

**Time**: 12-18 months
**Commitment**: 10 hours/week
**Output**: 8-10 working projects in your GitHub

**At the end of Phase 1, you should be able to**:
- Build production-grade Go services from scratch
- Design databases and write efficient queries
- Create well-designed APIs
- Understand distributed systems fundamentals
- Debug and troubleshoot complex issues independently

**If you can't do these, DO NOT move to Phase 2.** Going forward without this foundation is wasting time.

## Phase 2: System Design & Architecture (12-18 months)

**Goal**: Understand architectural patterns and design systems end-to-end.

### What to Learn

#### 1. Architectural Patterns (3-4 months)

**Core patterns**:
- Monolith vs microservices (when to use each)
- Event-driven architecture
- CQRS & Event Sourcing
- Saga pattern (distributed transactions)
- API Gateway pattern
- Service mesh basics

**Resources**:
- [ ] Read: "Building Microservices" by Sam Newman
- [ ] Read: "Software Architecture: The Hard Parts" by Ford et al.
- [ ] Study: [microservices.io patterns](https://microservices.io/patterns/)
- [ ] Watch: [Martin Fowler's talks on architecture](https://martinfowler.com/architecture/)

**Practice projects**:
1. Take one of your Phase 1 projects and break it into microservices (don't just split — understand WHY)
2. Implement event-driven communication between services
3. Implement CQRS pattern for a specific use case
4. Document architectural decisions (use ADRs - Architecture Decision Records)

**Success criteria**:
- Can explain when to use each pattern
- Understand trade-offs (not just benefits)
- Can diagram architectures clearly
- Know when NOT to use fancy patterns

#### 2. Platform & Infrastructure Architecture (4-6 months)

Since you want Platform Architect role, focus here.

**Core concepts**:
- Platform thinking: internal developer platforms
- Multi-tenancy patterns
- Infrastructure as Code (deep, not surface level)
- Service mesh (Istio/Linkerd)
- Observability platforms
- Security: zero trust, secrets management, least privilege
- Cost optimization & FinOps

**Resources**:
- [ ] Read: "Team Topologies" by Skelton & Pais — understand platform teams
- [ ] Study: [CNCF Landscape](https://landscape.cncf.io/) — understand the ecosystem
- [ ] Read: [AWS Well-Architected Framework](https://aws.amazon.com/architecture/well-architected/) — principles apply everywhere
- [ ] Study: [Backstage docs](https://backstage.io/) — modern platform thinking

**Practice projects**:
1. Design a multi-tenant platform architecture (draw diagrams, write docs)
2. Build a simple internal developer platform (API + CLI + docs)
3. Implement zero-trust security model for your services
4. Set up full observability stack (Prometheus, Grafana, Loki, Tempo)
5. Calculate and optimize cost for your infrastructure

**Success criteria**:
- Can design platform architecture for 50+ engineers
- Understand multi-tenancy patterns deeply
- Can make build vs buy decisions with justification
- Know security best practices
- Can design for cost efficiency

#### 3. System Design Practice (3-4 months)

**Core skill**: Given requirements, design a system end-to-end.

**What to practice**:
- Gathering requirements & constraints
- Capacity estimation (back-of-envelope calculations)
- High-level design
- Deep dive into components
- Trade-off analysis
- Scalability & reliability patterns

**Resources**:
- [ ] Read: "System Design Interview" by Alex Xu (both volumes)
- [ ] Study: [System Design Primer](https://github.com/donnemartin/system-design-primer)
- [ ] Practice: Design real systems your company uses

**Practice problems** (design these end-to-end):
1. URL shortener (start simple, scale to 100M URLs/day)
2. Rate limiter (multiple algorithms, distributed)
3. Notification system (email, SMS, push)
4. Distributed cache
5. API gateway
6. Logging/monitoring system
7. CI/CD platform
8. Kubernetes operator platform (relevant to your work)

**For each design**:
- Write a design doc (requirements, constraints, high-level design, deep dive, trade-offs)
- Draw architecture diagrams
- Calculate capacity needs
- Identify failure modes
- Explain your decisions

**Success criteria**:
- Can design systems end-to-end in 45-60 minutes
- Can explain trade-offs clearly
- Can estimate capacity needs
- Can identify and handle failure modes
- Can communicate designs through docs and diagrams

#### 4. Networking & Security Deep Dive (2-3 months)

You said you're interested in this, so now's the time.

**Core concepts**:
- OSI model & TCP/IP (beyond basics)
- DNS, load balancing, CDN
- TLS/SSL, certificates, mutual TLS
- Network policies in Kubernetes
- Zero trust networking
- Common vulnerabilities (OWASP Top 10)
- Security controls & defense in depth

**Resources**:
- [ ] Read: "Computer Networking: A Top-Down Approach" (Kurose & Ross) — chapters 1-6
- [ ] Course: [Cybrary Network+ (free sections)](https://www.cybrary.it/)
- [ ] Study: [OWASP Top 10](https://owasp.org/www-project-top-ten/)

**Practice projects**:
1. Build a simple load balancer in Go
2. Implement mutual TLS between services
3. Set up network policies in K8s cluster
4. Perform security audit of one of your projects
5. Fix vulnerabilities (SQL injection, XSS, etc.)

**Success criteria**:
- Can troubleshoot network issues (DNS, routing, firewall)
- Understand TLS/mTLS deeply
- Can design secure network architecture
- Know common vulnerabilities and how to prevent them
- Can perform basic security reviews

## Phase 3: Architecture Transition (12+ months)

**Goal**: Start doing architecture work and transition your role.

### What to Do

#### 1. At Your Current Job (Critical)

**Start behaving like an architect before you have the title**:

**Months 1-3: Observe & Document**
- Document the architecture of systems you work on
- Create architecture diagrams for your team's services
- Write ADRs (Architecture Decision Records) for decisions your team makes
- Ask "why" questions in every design discussion

**Months 4-6: Small Design Decisions**
- Volunteer to design small features end-to-end
- Write design docs for your proposed solutions
- Present designs to your team
- Incorporate feedback and iterate

**Months 7-9: Larger Scope**
- Propose architecture improvements for existing systems
- Lead technical discussions
- Review others' designs and give feedback
- Mentor junior engineers

**Months 10-12: Cross-team Impact**
- Design solutions that span multiple teams
- Collaborate with other architects
- Present to senior leadership
- Build your reputation as technical leader

**Key skills to practice**:
- Writing clear design docs
- Creating understandable diagrams
- Presenting technical concepts to non-technical stakeholders
- Facilitating technical discussions
- Building consensus
- Making decisions with incomplete information

#### 2. Build Your Portfolio

**Create a public portfolio** showing:
- System designs you've done (anonymize if needed)
- Architecture documents
- Technical blog posts explaining your decisions
- Open source contributions
- Speaking at meetups/conferences

**Why**: When you interview for architect roles, this proves you can do the job.

#### 3. Learn the Business Side

**Architects must understand business context**:
- Learn about your company's business model
- Understand cost implications of technical decisions
- Connect technical work to business outcomes
- Learn to speak the language of product and business

#### 4. Network & Learn from Others

- Find an architect mentor (inside or outside your company)
- Join architecture communities (Software Architecture Discord, architecture meetups)
- Review open source projects' architecture
- Read postmortems from other companies
- Study how big tech companies solve problems (engineering blogs)

#### 5. Communication & Documentation Skills

**This is 70% of the architect job**:
- Practice writing: start a technical blog
- Practice speaking: present at team meetings, then meetups
- Learn diagramming: C4 model, UML basics
- Study good documentation: read RFCs, design docs from big companies
- Get feedback on your communication constantly

### Phase 3 Summary

**Time**: 12+ months
**Commitment**: 10 hours/week + applying learning at work
**Output**: Portfolio of designs, blog posts, presentations

**At the end of Phase 3, you should**:
- Be doing architecture work at your current job
- Have a portfolio demonstrating your skills
- Be ready to interview for architect roles
- Have a network in the architecture community

## The Final Truths

### What Makes an Architect (The Real Definition)

An architect is someone who:
1. **Understands the problem deeply** (not just the solution)
2. **Makes informed trade-off decisions** (not perfect decisions)
3. **Communicates designs clearly** (to engineers and non-engineers)
4. **Learns from failures** (their own and others')
5. **Knows when to say "I don't know"** (and finds the answer)
6. **Builds consensus** (not just imposes solutions)
7. **Thinks long-term** (not just quick wins)
8. **Codes enough to stay credible** (but not full-time)

**Notice what's NOT on this list:**
- Knowing every technology
- Having all the answers immediately
- Being the best coder
- Working alone
